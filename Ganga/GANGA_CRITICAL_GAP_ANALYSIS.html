Project Ganga — Critical Gap Analysis

Opus 4.7 Deep Analysis · 2026-05-04 · MARSYS-JIS Platform Elevation Sprint
⚠ Opus Verdict

"Project Ganga as currently scoped will produce a robust five-stack RAG platform with reliable planner and observable retrieval. It will not produce an acharya-grade Jyotish instrument. The synthesis layer — the only layer that touches the native's actual question — has zero coverage in the current scope."

5 Critical Missing Elements
GAP 013
Synthesis Prompt Architecture — Zero governance
The synthesis model (claude-opus-4-6) receives no Jyotish-specific instruction. It has no citation requirements, no B.11 Whole-Chart-Read enforcement, no per-query-class behavior, no counterfactual discipline. It produces whatever an intelligent LLM produces by default — which is not acharya-grade.
  • No SYNTHESIS_PROMPT_v1_0.md exists (analogous to PLANNER_PROMPT_v1_0.md)
  • No B.3 derivation-ledger discipline encoded in synthesis instructions
  • No per-class variants (predictive ≠ interpretive ≠ remedial ≠ holistic)
  • No counterfactual check instruction ("does this claim survive if X is reversed?")
  • No output schema (structured sections vs free-form prose)
GAP 014
Answer Quality Evaluation — No metric, no harness, no CI gate
The planner has recall=0.940 and precision=0.945 because it was rigorously evaluated across 6 rounds and a 29-case golden set. The synthesizer has no metric at all. This means quality regressions in synthesis are completely invisible — there is no way to know if a synthesis prompt change made answers better or worse.
  • No 100-query acharya-grade answer eval set
  • No LLM-as-judge harness (citation_coverage, b11_compliance, acharya_depth_score)
  • No baseline score (no "before" state to measure against)
  • No CI gate — synthesis quality regressions ship silently
  • No multi-stack comparison (which stacks produce better answers?)
GAP 015
B.11 Whole-Chart-Read — No runtime enforcement
CLAUDE.md §I names B.11 (Whole-Chart-Read Protocol) as one of the five most-violated principles: "A query-answer that skips L2.5 consultation is a procedural violation equivalent to a red-team finding." The planner can currently produce plans that call only one tool, skipping MSR/UCN/CDLM/CGM entirely. No gate exists to catch this at runtime.
  • No holistic floor gate in UQE (no minimum L2.5 layer requirement)
  • No post-synthesis citation audit (does the answer actually cite MSR signal IDs?)
  • No B.11 violation counter in AnalyticsTab
GAP 016
Context Assembly Layer — Missing entirely
Multiple tools return results independently. Nothing assembles them into a coherent context before synthesis. No deduplication (MSR SQL and vector_search may return overlapping signals), no Jyotish-methodology-aware ordering (FORENSIC floor must be first), no conflict surfacing (two tools making contradictory claims about the same house/planet).
  • No context_assembler.ts module
  • FORENSIC data not guaranteed as mandatory floor (ADR-4 enforcement gap)
  • No conflict detection between tool outputs
  • No compression for long contexts (token budget management)
GAP 017
Calibration Substrate — One item, no wiring
The entire learning dimension of the project — the mechanism by which the instrument improves over time based on real-world prediction outcomes — is represented by one line item in "adjacent systems." The synthesis pipeline has no prediction extraction hook, outcome capture is not linked to synthesis answers, and Brier scores cannot be computed.
  • No prediction registration hook in synthesis pipeline (L.5.1 missing)
  • Phase 8 outcome form exists but not linked to prediction_outcome_log
  • No Brier score computation per query_class
  • No native correction surface feeding back to synthesis prompt revision
