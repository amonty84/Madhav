# STEP LEDGER v1.0 — Governance Rebuild Workflow (Steps 0 → 15, with Step 5A inserted)

## How to use this file

1. A fresh conversation that intends to execute **Step N** first reads this ledger.
2. It confirms: Step N is `pending`, Step N-1 is `completed`, and no other step is `in_progress`.
3. If any condition fails, the session halts and reports to the native instead of "improvising".
4. On successful execution, the session updates the row to `completed`, fills in the deliverable path, appends a history entry at the bottom, and hands off.
5. **At session open, the session SHOULD propose the conversation's Cowork-thread name per `00_ARCHITECTURE/CONVERSATION_NAMING_CONVENTION_v1_0.md` §4** — emitted at the top of the first substantive response. The proposed name is derived from whichever row is `in_progress` (or, if none, the next `ready` row). This is a LIVING bridge rule until Step 9 rebuilds CLAUDE.md and absorbs the session-open instruction there.

Allowed status values:
- **pending** — not yet started; prerequisites may or may not be met
- **ready** — prerequisites met; next conversation can pick this up
- **in_progress** — a session is currently executing this step
- **blocked** — a flag has been raised; see `blocking_notes` column
- **completed** — closed, deliverable exists, red-team (if due) passed
- **superseded** — workflow revised; step no longer applies

## Blocking-flag conventions

If a step encounters a condition it cannot handle without native input, it sets status = `blocked` and writes a `blocking_notes` entry. A `blocked` step does not count as `in_progress`. Resolving the block is a conversation with the native, not another step execution.

And updates the ledger row accordingly. The next conversation begins by reading this ledger, this handoff note, and the Step N+1 brief — nothing else is required.
