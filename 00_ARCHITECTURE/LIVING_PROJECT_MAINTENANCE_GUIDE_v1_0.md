## §1 — MAINTENANCE CADENCE

### §1.1 — Quarterly maintenance (every 3 months)

**Scope**: HEATMAP refresh and near-term timing-window updates.

**Actions**:
1. Re-read HEATMAP_VARSHPHAL_v1_0
2. Check current dasha PD (Pratyantara-dasha) for the specific quarter
3. Check current transit positions against natal degrees
4. Note any significant transit activations in the quarter
5. Update if material changes warrant new entries

**Time required**: 2-3 hours per quarterly check.

### §1.2 — Annual maintenance (every year, recommended near birthday)

**Scope**: Varshphal refresh, Life Event Log bump, Heatmap refresh.

**Actions**:
1. Compute Varshphal for the new solar year (use Jagannatha Hora)
2. Update LIFE_EVENT_LOG with significant events of past year
3. Re-test retrodictive fit of past-year events against chart predictions
4. Update HEATMAP_VARSHPHAL for upcoming year
5. Review MSR for signals that may have shifted in confidence based on new empirical data
6. Update SESSION_LOG with annual-refresh entry

**Time required**: 8-12 hours per annual refresh (full-day session).

### §1.4 — Major-life-event triggered refresh

Beyond scheduled maintenance, the following life events trigger immediate refresh:
- Major career change (new role, new company, exit)
- Major relational event (commitment, dissolution, significant conflict)
- Major health event (serious diagnosis, major intervention)
- Major financial event (large loss, large gain, commitment beyond threshold)
- Major location change (relocation, extended travel beyond 3 months)
- Major spiritual event (significant practice change, teacher change)

**Action**: Run a focused Query (per Query Prompt Library template for that decision-type). Update relevant Domain Report(s) if findings warrant.

## §3 — VERSION-BUMP TRIGGERS

### §3.1 — When to bump minor version (v1.0 → v1.1)

- Factual error correction
- New finding that refines but does not overturn
- Known gap resolution (e.g., v6.0 → v7.0 for Facts Layer)
- Red-team-identified refinement

### §3.2 — When to bump major version (v1.0 → v2.0)

- Architectural change (with native approval)
- Major new synthesis concept meriting UCN incorporation

- Post-external-acharya-review with substantive feedback
